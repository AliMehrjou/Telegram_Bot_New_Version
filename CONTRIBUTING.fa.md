# مشارکت در توسعه ربات دیتینگ تلگرام

از اینکه در این پروژه مشارکت می‌کنید سپاسگزاریم! این سند شامل دستورالعمل‌ها و مروری بر معماری پروژه است تا به شما در درک نحوه کارکرد ربات کمک کند و اطمینان حاصل شود که توسعه‌های آینده پایدار، غیرهمزمان (Async) و مقیاس‌پذیر باقی می‌مانند.

## 🏗 مروری بر معماری

این ربات بر پایه یک معماری کاملاً غیرهمزمان (Asynchronous) طراحی شده است تا از مسدود شدن (Blocking) فرآیندها و مشکلات ناشی از همزمانی (Race Conditions) جلوگیری کند.

- **FastAPI و aiogram:** سرور FastAPI وظیفه دریافت درخواست‌های وب‌هوک (Webhook) از تلگرام و ارائه یک API مدیریت (Admin API) داخلی را بر عهده دارد. به‌روزرسانی‌های دریافتی به صورت غیرهمزمان به Dispatcher (`dp`) فریم‌ورک `aiogram` ارسال می‌شوند.
- **سلسله‌مراتب میان‌افزارها (Middlewares):** میان‌افزارها به ترتیب زیر اجرا می‌شوند تا قوانین کلی پیش از رسیدن درخواست به Handlerها اعمال شوند:
  1. `ThrottlingMiddleware`: جلوگیری از اسپم (Spam).
  2. `DbSessionMiddleware`: تزریق یک `AsyncSession` فعال به Handlerها و به‌روزرسانی وضعیت آنلاین (`is_online`) کاربران.
  3. `ForceJoinMiddleware`: الزام کاربران به عضویت در کانال اجباری. این میان‌افزار در صورتی که کاربر در وضعیت فعال Redis (مثل `matched` یا `chatting`) باشد، بررسی را نادیده می‌گیرد تا جلسات زنده (Live Sessions) متوقف نشوند.

## 🗄 پایگاه داده و مدل‌ها (`models.py`)

ما از **SQLAlchemy 2.0 (Async)** استفاده می‌کنیم. تمام مدل‌ها و جداول پایگاه داده در مسیر `database/models/models.py` قرار دارند.

### بهترین روش‌ها برای کار با مدل‌ها:
- **به‌روزرسانی مدل‌ها:** هنگام افزودن یک ستون جدید (مانند `city` یا `tags`)، همیشه آن را با استفاده از استاندارد SQLAlchemy 2.0 یعنی `Mapped[Type] = mapped_column(...)` تعریف کنید.
- **مایگریشن‌ها (Migrations):** ما از Alembic استفاده می‌کنیم. هر زمان که تغییری در `models.py` ایجاد کردید، حتماً یک اسکریپت مایگریشن جدید تولید کرده و دیتابیس را به‌روز (Upgrade) کنید.
- **ارتباطات (Relationships):** ارتباطات کلید خارجی (`ForeignKey`) را به صورت دقیق با استفاده از `ondelete="CASCADE"` یا `SET NULL` (بسته به منطق داده‌ها) تنظیم کنید تا از وجود رکوردهای یتیم (Orphaned Records) جلوگیری شود.

## 📝 سبک کدنویسی و استانداردها

برای حفظ سلامت کد، به شدت از قوانین زیر پیروی کنید:

1. **اجرای کاملاً غیرهمزمان (Async):**
   - **هرگز** از متد همزمان `time.sleep()` استفاده نکنید. همیشه از `await asyncio.sleep()` استفاده کنید.
   - تمام عملیات دیتابیس باید از `AsyncSession` و `await session.execute(...)` استفاده کنند.

2. **اقتصاد ایمن (تراکنش‌های اتمیک):**
   - تراکنش‌هایی که `coin_balance`، `total_earned_coins` یا `total_spent_coins` را تغییر می‌دهند، باید به صورت کامل ثبت (Commit) شوند.
   - در بخش مچینگ، کسر سکه برای جلوگیری از کسر شدن اشتباه (Ghost Deductions)، **تنها پس از تایید اتصال موفق** انجام می‌شود.

3. **مدیریت ایمن API تلگرام:**
   - هرگونه فراخوانی API خارجی (مثل `bot.send_message` و `bot.copy_message`) در صورت مسدود شدن ربات یا حذف اکانت کاربر ممکن است با خطا مواجه شود.
   - **همیشه** فراخوانی‌های API را در داخل بلاک‌های `try/except` قرار دهید تا خطاهای `TelegramForbiddenError` و `TelegramAPIError` گرفته شده و از توقف حلقه رویدادها (Event Loop) جلوگیری شود.

4. **مدیریت FSM بینابینی کاربران (Cross-User FSM):**
   - برای خواندن یا تغییر وضعیت (State) کاربر دیگر (پارتنر)، وضعیت آن‌ها را به صورت پویا (Dynamic) ایجاد کنید:
     ```python
     ctx = FSMContext(storage=dp.storage, key=StorageKey(bot_id=bot.id, chat_id=partner_id, user_id=partner_id))
     await ctx.clear()
     ```

## 🔄 جریان کاری درخواست ادغام (Pull Request Workflow)

1. **ایجاد شاخه (Branching):** برای قابلیت جدید یا رفع باگ خود یک شاخه جدید بسازید (مانند `feature/add-gift-system` یا `bugfix/fix-timeout`).
2. **تست محلی:** مطمئن شوید فایل `.env` به درستی تنظیم شده است. ربات را روی سیستم خود تست کنید. بررسی کنید که تغییرات دیتابیس به درستی اعمال شده باشند و باگ جدیدی اضافه نشده باشد.
3. **پیام کامیت (Commit Messages):** پیام‌های کامیت دقیق و مشخص بنویسید که دلیل و تغییرات انجام شده را توضیح دهند.
4. **ارسال PR:** یک Pull Request به شاخه `main` ارسال کنید. در توضیحات آن خلاصه‌ای از تغییرات ارائه دهید و مشکلات مرتبط را تگ کنید.
